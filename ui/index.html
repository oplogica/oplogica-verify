Synthetic medical triage scenario. Not clinical advice. Not a medical device. Not validated for clinical use. Data is illustrative and fabricated for demo purposes. Demo keys only — not HSM-backed, not production PKI.
Scope. This demo verifies evidence integrity and binding only: whether the recorded artifacts are authentic, internally consistent, and untampered under the demo trust root. It does not establish decision correctness, model behavior fairness, or legal readiness, and is not intended for critical deployment. The conflict check confirms that recorded conflicts stay visible and intact — it does not detect every possible conflict. The reasoning checks confirm the recorded reasoning graph is well formed, authenticated, and bound to the policy and operation — they do not establish reasoning quality or formal entailment.

Verify a bundle

Paste a bundle, governance registry, and trust root (and optionally the raw input data), then verify offline through the API. Use Load demo clean inputs to populate the fields, or load them from the generated exports/ files.

Demo inputs are read from ../exports/ (run the pipeline first: python3 ova_demo/run_demo.py --out ./exports). If they are not found, paste JSON manually.

Tamper demonstration — T4 (reasoning graph)

T4 adds a non-existent reference (p_GHOST) to a reasoning conclusion. One narrow mutation produces two failed checks and causes two more checks not to run. Use Apply T4 tamper to bundle above, then Verify, to see it live.

Why T4 fails

por_structural_consistency — the conclusion references a node that does not exist, so the recorded reasoning graph is malformed. (This does not establish anything about whether the reasoning was good — only that the recorded structure is broken.)

merkle_root_match — changing the reasoning layer changes its hash, so the recomputed Merkle root no longer matches the stored root.

Why two checks do not run

por_signature_binding_valid and por_rule_policy_binding sit inside the same reasoning block, after the structural gate. When the structural check fails, the engine short-circuits past them.

The reconciliation layer reports them by name as not run, with a reason — instead of letting them quietly disappear. Passed + failed + not run always sums to 13.

Evidence Integrity Report

The full human-readable report is generated to exports/evidence_integrity_report.md.

Open it directly: ../exports/evidence_integrity_report.md  ·  or render it as Markdown in any viewer.

The report includes: the clean-bundle summary with the meaning / not-meaning block, a summary table of all six tamper cases, a per-case detail section with failed and not-run checks and their explanations, reproduction commands, and a limitations section.

Reproduce from the command line

Run the full demo

python3 ova_demo/run_demo.py --out ./exports

Verify the clean bundle (expect VALID, exit 0)

python3 ova_demo/verify_bundle.py exports/clean_bundle.json \
  --registry exports/registry.json \
  --trust-root exports/trust_root.json \
  --input-data exports/input_data.json

Verify the T4 tampered bundle (expect INVALID, exit 1)

python3 ova_demo/verify_bundle.py exports/tampered_T4_por.json \
  --registry exports/registry.json \
  --trust-root exports/trust_root.json \
  --input-data exports/input_data.json

Exit codes for verify_bundle.py: 0 = reconciled VALID, 1 = reconciled INVALID, 2 = usage/file/JSON error.

Reference implementation in Python. Go/Rust SDKs do not exist. Signatures are Ed25519-prototype-2026 (not post-quantum); the production target ML-DSA (Dilithium-III, FIPS 204) is declared but not implemented. The Merkle tree is a simple binary tree and does not implement the RFC 6962 construction.
